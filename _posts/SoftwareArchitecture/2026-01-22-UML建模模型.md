## 1. 概述
软件体系结构模型应具备多维特性，而非单一结构。只有将以下五种模型有机结合，才能全面刻画软件的框架结构。

![软件体系结构模型](/img/SoftwareArchitecture/软件体系结构模型.png)

### 1.1 UML简介
1. UML描述内容——**4+1视图**（逻辑视图、开发视图、过程视图、物理视图、场景视图）
![4+1](/img/SoftwareArchitecture/4+1视图.png)

2. UML（统一建模语言）定义：
- UML是一种拥有标准表达规则的建模语言，区别于Java、C++等编程语言。它用于分析和设计，是一种<font color=green>建模语言</font>。
- UML通过<font color=amber>图形符号</font>进行表达。
- UML是用于<font color=camel>描绘软件蓝图</font>的标准语言。

### 1.2 UML分类

#### 1.2.1 按图表类型划分

| 序号 | UML图                                     | 简要说明                                                                                       |
| ---- | ----------------------------------------- | ---------------------------------------------------------------------------------------------- |
| 1    | 用例图（UseCase Diagram）                 | 从用户角度概述系统或业务流程功能，展示用户与系统的交互，是建模的起点。                         |
| 2    | 活动图（Activity Diagram）                | 建模系统中各处的流程，尤其适用于描述用例中的正常、替代及异常流程。                             |
| 3    | 类图（Class Diagram）                     | 展示类的定义及其关系，描述类的属性、操作及其交互、协作、继承等。也可表示关系表、界面和控制器。 |
| 4    | 序列图（Sequence Diagram）                | 按时间轴模拟对象间的交互，展示对象之间消息的执行顺序。                                         |
| 5    | 交互概述图（Interaction Diagram）         | 高层次呈现系统内的交互，帮助理解各UML图之间的依赖和关联。                                      |
| 6    | 通信图（Communication Diagram）           | 展示对象在运行时的通信方式，与序列图类似但表达方式不同。                                       |
| 7    | 对象图（Object Diagram）                  | 展示运行时内存中的对象及其链接，便于可视化多重性。                                             |
| 8    | 状态机图（State Machine Diagram）         | 展示对象在运行时的生命周期，包括所有状态及状态转换条件。                                       |
| 9    | 组合结构图（Composite Structure Diagram） | 模拟组件或对象在运行时的行为，展示系统执行期间组件的布局、关系和实例。                         |
| 10   | 组件图（Component Diagram）               | 建模组件及其关系，包括可执行文件、库、服务等，为架构决策提供支持。                             |
| 11   | 部署图（Deployment Diagram）              | 建模系统硬件节点和处理器架构，展示软件组件的部署位置。                                         |
| 12   | 包图（Package Diagram）                   | 展示系统的子系统和区域，模拟包之间的依赖关系，便于业务实体与界面、数据库、安全等分离。         |
| 13   | 时序图（Timing Diagram）                  | 模拟时间概念，展示对象状态随时间变化，并可比较多个对象的状态。                                 |
| 14   | 配置文件图（Profile Diagram）             | 创建可扩展的配置文件，受控扩展标准元素，提升建模灵活性。                                       |

#### 1.2.2 按视图类型划分

- **结构视图**（描述系统静态结构）
  - **类图**
  - **包图**
  - 对象图（类图的实例）
  - 组合结构图（描述复杂类的内部结构，如内部类）

- **实现视图**（描述系统实现后文件间关系）
  - **组件图**

- **行为视图**（描述系统运行时元素的交互）
  - **顺序图**（描述次序）
  - 通信图（描述关系，顺序图与通信图可相互转换）
  - 定时图
  - **状态图**
  - **活动图**（支持泳道）
  - 交互概览图（细化活动图中的每个活动，使用频率较低）

- **环境视图**（描述系统实现后硬件间关系）
  - **部署图**

- **用户视图**
  - **用例图**

### 1.3 UML的结构
1、**模型元素**(Model element)
- 模型元素包括事物以及事物与事物之间的联系
- 每一个模型元素都有一个与之相对应的图形元素
- 同一个模型元素可以在不同的UML图中使用
- 无论在哪个图中，<font color=brown>同一个模型元素都保持相同的意义和符号</font>

2、**通用机制**(General mechanism)
- 额外的注释、修饰和语义等
- 包括规格说明、修饰、公共分类和扩展机制四种
- 允许用户对UML进行扩展

### 1.4 UML的特点
工程化；规范化；可视化；系统化；文档化；智能化

## 2. UML图表
### 2.1 类图
#### 2.1.1 定义
- 类图使用需要出现在系统内的不同的类来描述系统的**静态结构**，类图包含**类**和它们之间的**关系**，它描述系统内所声明的类，但它<font color=tomato>没有描述系统运行时类的行为</font>。
- 在UML中，类使用具有类名称、属性、操作分隔的长方形来表示：
![类图例子](/img/SoftwareArchitecture/类图例子.png)

#### 2.1.2 组成
类由三格组成：
- 第一格是**类名**。
- 第二格是类的**属性**—— 可见性 名称:类型 [= 默认值]。
- 第三格是类的**操作**—— 可见性 名称([参数列表]) [: 返回类型]。

#### 2.1.3 关系
1、**关联关系**
- 关联关系(Association)是类与类之间最常用的一种关系，它是一种结构化关系，用于表示一类对象与另一类对象之间有联系。
- 在UML类图中，用实线连接有关联的对象所对应的类，在使用Java、C#和C++等编程语言实现关联关系时，**通常将一个类的对象作为另一个类的属性**。
- 在使用类图表示关联关系时**可以在关联线上标注角色名**。
![关联关系](/img/SoftwareArchitecture/关联关系.png)
- 双向关联：没有箭头
- 自关联：绕一圈指向自己（单例模式常见）
- 多重性关联：在上面增加两个m..n，注意表达的含义别反了
![多重性关联](/img/SoftwareArchitecture/多重性关联.png)

| 表示方式 | 多重性说明                                                 |
| -------- | ---------------------------------------------------------- |
| 1..1     | 表示另一个类的一个对象只与一个该类对象有关系               |
| 0..*     | 表示另一个类的一个对象与零个或多个该类对象有关系           |
| 1..*     | 表示另一个类的一个对象与一个或多个该类对象有关系           |
| 0..1     | 表示另一个类的一个对象没有或只与一个该类对象有关系         |
| m..n     | 表示另一个类的一个对象与最少m、最多n个该类对象有关系 (m≤n) |

2、**聚合关系**
- 聚合关系(Aggregation)表示一个**整体与部分的关系**。通常在定义一个整体类后，再去分析这个整体类的组成结构，从而找出一些成员类，该整体类和成员类之间就形成了聚合关系。 
- 在聚合关系中，**成员类是整体类的一部分**，即成员对象是整体对象的一部分，但是**成员对象可以脱离整体对象独立存在**（<font color=green>主要区别</font>）。在UML中，聚合关系用带空心菱形的直线表示。 
![聚合关系](/img/SoftwareArchitecture/聚合关系.png)

3、**组合关系**
- 组合关系(Composition)也表示类之间**整体和部分的关系**，但是组合关系中部分和整体具有统一的生存期。**一旦整体对象不存在，部分对象也将不存在**（<font color=green>主要区别</font>），部分对象与整体对象之间具有同生共死的关系。
- 在组合关系中，成员类是整体类的一部分，而且整体类可以控制成员类的生命周期，即成员类的存在依赖于整体类。在UML中，组合关系用带实心菱形的直线表示。
![组合关系](/img/SoftwareArchitecture/组合关系.png)

4、**依赖关系**
- 依赖关系(Dependency)是一种使用关系，特定事物的改变有可能会影响到使用该事物的其他事物，在需要表示一个事物使用另一个事物时使用依赖关系。大多数情况下，依赖关系体现在**某个类的方法使用另一个类的对象作为参数**（方法参数或者new()）。
- 在UML中，依赖关系用带箭头的虚线表示，由依赖的一方指向被依赖的一方。
![依赖关系](/img/SoftwareArchitecture/依赖关系.png)

5、**泛化关系**
- 泛化关系(Generalization)也就是**继承关系**，也称为“is-a-kind-of”关系，泛化关系用于描述父类与子类之间的关系，父类又称作基类或超类，子类又称作派生类。在UML中，泛化关系用带空心三角形的直线来表示。
- 在代码实现时，使用面向对象的继承机制来实现泛化关系，如在Java语言中使用extends关键字、在C++/C#中使用冒号“：”来实现。 
![泛化关系](/img/SoftwareArchitecture/泛化关系.png)

6、**实现关系** 
- 接口之间也可以有与类之间关系类似的继承关系和依赖关系，但是接口和类之间还存在一种实现关系(Realization)，在这种关系中，类实现了接口，类中的操作实现了接口中所声明的操作。在UML中，类与接口之间的实现关系用带空心三角形的虚线来表示。 
![实现关系](/img/SoftwareArchitecture/实现关系.png)

### 2.2 包图

#### 2.2.1 定义
- **包**是一种**把元素组织到一起**的通用机制，包可以嵌套于其他包中。
- **包图**用于描述**包与包之间的关系**，包的图标是一个带标签的文件夹。
![包](/img/SoftwareArchitecture/包.png)

#### 2.2.2 关系
1、**引入关系**：一个包中的类可以被另一个指定包（以及<font color=tomato>嵌套于其中的那些包</font>）中的类引用。（相当于include）
![引入关系](/img/SoftwareArchitecture/包引入关系.png)
2、**泛化关系**：表示一个包继承了另一个包的内容，同时又补充自己增加的内容。
![泛化关系](/img/SoftwareArchitecture/包泛化关系.png)
3、**嵌套关系**：一个包中可以包含若干个子包，构成了包的嵌套层次结构。（相当于子系统），下面是箭头样式，要画成这种，比较简单不多说了
![嵌套关系](/img/SoftwareArchitecture/嵌套关系.png)

### 2.3 组件图

#### 2.3.1 定义
**组件图**又称为**构件图**(Component Diagram) 。组件图中通常包括<font color=seagreen>组件、接口，以及各种关系</font>。组件图显示组件以及它们之间的**依赖关系**，它可以用来显示程序代码如何分解成模块或组件（常用于<font color=tomato>显示编译、链接或执行时组件之间的依赖关系，以及组件的接口和调用关系</font>）。

#### 2.3.2 关系：泛化和依赖
#### 2.3.3 组成
- **组件**：系统中可以替换的部分，一般对应一个实际文件，如exe、jar、dll等文件，它遵循并提供了一组接口的实现。
- **接口**：一组操作的集合，它指明了由类或组件所请求或者所提供的服务，它是一个标准定义。
- **部件**：组件的局部实现。
- **端口**：被封装的组件与外界的交互点，遵循指定接口的组件通过它来收发消息。
- **连接件**：在特定语境下组件中两个部件之间或者两个端口之间的通信关系。
![组件图例子](/img/SoftwareArchitecture/组件图例子.png)

#### 2.3.4 绘制技巧
- 当需要把系统分成若干组件（构件），希望借助接口或组件将系统分解为低层结构并表示其相互关系时需要使用组件图。
- 在绘制组件图时，应该注意侧重于描述系统的静态实现视图的一个方面，图形不要过于简化，应该为组件图取一个直观的名称，在绘制时避免产生线的交叉。
- 注意组件的粒度，粒度过细的构件将导致系统过于庞大，会给版本管理带来问题。
