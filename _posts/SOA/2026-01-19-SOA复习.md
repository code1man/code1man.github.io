## 1. SOA概述

### 1.1 引入SOA的背景与待解决的挑战

**原因**  
- <font color=purple>需求驱动</font>：互联网环境下企业之间的协同、异构系统之间的交互与互操作需求增加。  
- <font color=brown>技术推动</font>：云计算、虚拟化、面向服务的架构与SOD的引入。

> PS：互操作性（Interoperability）指异构分布式系统之间交换和使用信息的能力。

**SOA要应对的问题**  
- 分布式企业业务协同。  
- 异构应用系统间的集成、交互与互操作。  
- 业务发生变化时IT系统快速响应的能力。  
- 四大目标：屏蔽异构性、实现互操作、提炼共性复用、构建柔性可变系统。

### 1.2 SOA的定义与组成要素

**SOA定义**：Service（服务）+ Architecture（体系结构）

**三大核心要素**：标准化封装、复用、松耦合可编排（目标是高内聚、低耦合的灵活系统）  
![SOA核心要素](/img/SOA/SOA核心要素.png)

**基本构件与连接件**  
- **服务**：显式的、独立于实现技术的接口；封装可复用的大粒度业务功能（业务过程/活动）。  
- **连接件**：接口（如 RESTful API）。  

> PS：SOA 中所有协议均基于 XML 的文本格式。

**本质特征**  
- 以自治、平台独立、可描述发布发现及动态组装的服务作为构造单元。  
- 应用被视为协同运作的一组服务。  
- 快速、低成本、易组合地创建高度分布式、协同、动态、跨组织与平台边界的服务系统。

### 1.3 SOA的典型优势

- 支持不同<font color=green>平台</font>、<font color=chocolate>语言</font>与<font color=plum>架构</font>之间的集成与互操作。  
- 实现完全的松散耦合：位置透明、独立于实现（接口调用）、使用标准化通信协议（XML-based）。  
- 适用于<font color=blue>大数据量低频访问</font>场景。  
- 基于文本的消息传递，避免兼容性问题。  
- 服务<font color=tomato>独立于具体使用者上下文</font>：可复用性高。  
- 强调业务过程/活动级别的<font color=seagreen>大粒度复用</font>。  
适用于<font color=seagreen>协同、交互、异构、分布式与频繁变化</font>的环境。

### 1.4 SOA体系结构模式

1. 硬连线（Hard-wired）  
2. 点对点服务发布与调用（P2P）  
3. 服务适配器（Service adaptor）  
4. 服务代理（Service proxy）  
5. 远程服务策略（Remote service strategy）  
6. 单点访问（Single point of access）  
7. 虚拟服务提供者（Virtual provider）  
8. 服务集成器（Service integrator）  
9. 企业服务总线（Enterprise service bus）  
10. 集成化服务生态系统（Integrated service ecosystem）

#### 1.4.1 发布-查询-绑定模式（P2P）

- 场景：Web（详见 2 章）  
- 基本模式：发布 → 查询 → 绑定  
![P2P](/img/SOA/P2P.png)

#### 1.4.2 服务适配器模式（Service Adaptor/Wrapper）

用于复用老系统，类似架构中的适配器角色。

#### 1.4.3 远程服务策略

- **问题**：调用服务需要在代码中写死 URI，耦合度高、灵活性弱。  
- **解决**：将服务地址等配置信息放到配置文件，例如 service registry，降低耦合。

#### 1.4.4 服务集成器

- 将多个小粒度服务按照逻辑组合成一个集成服务。  
- 客户端调用一个服务即可；集成器根据配置依次调用各个服务。  
- 修改配置即可实现快速变更。

#### 1.4.5 企业服务总线（ESB）

**概念**：支持应用与服务公平整合的灵活连接基础设施，便于服务编排。

**ESB实现能力**  
1. 路由服务间的消息  
2. 转换请求者与服务之间的传输协议  
3. 转换请求者与服务之间的消息格式  
4. 处理分离资源之间的业务事件

## 2. Web服务基础

### 2.1 Web服务概念

- Web Service 是一个通过网络远程调用的、向外暴露接口的应用程序。  
- 它既是部署在 Web 上的对象，又是一套基于 XML 等开放 Web 标准的分布式应用平台，定义了应用在 Web 上互操作的方式。

#### 2.1.1 Web服务特性

具备可描述性、可发布性、可查找性、可绑定、可调用、可组合等特征。

#### 2.1.2 核心标准

![WebService核心标准](/img/SOA/WebService核心标准.png)

### 2.2 SOAP（Simple Object Access Protocol）

#### 2.2.1 SOAP的基本概念

1. 指“简易对象访问协议”。  
2. 是一种通信协议。  
3. 用于不同应用之间通信。  
4. 定义消息格式，不规定额外结构、编码或传输协议。  
5. 设计用于通过互联网通信。  
6. 平台无关。  
7. 语言无关。  
8. 基于 XML，可使用任意模式定义内部内容结构。  
9. 简单且可扩展。  
10. 可绕过防火墙。  
11. 作为 W3C 标准持续发展。

#### 2.2.2 SOAP消息结构

![SOAP消息结构](/img/SOA/SOAP消息结构.png)

基本格式：

```xml
<?xml version="1.0"?>
<soap:Envelope
  xmlns:soap="http://www.w3.org/2001/12/soap-envelope"
  soap:encodingStyle="http://www.w3.org/2001/12/soap-encoding">

  <soap:Header>
    ...
  </soap:Header>

  <soap:Body>
    ...
    <soap:Fault>
      ...
    </soap:Fault>
  </soap:Body>
</soap:Envelope>
```

| 部分     | 是否必需 | 作用                   |
| -------- | -------- | ---------------------- |
| Envelope | ✅        | 标识 SOAP 消息         |
| Header   | ❌        | 承载安全、认证、路由等 |
| Body     | ✅        | 包含业务请求/响应      |
| Fault    | ❌        | 返回错误信息           |

**类比**

- Envelope：信封外层 → 表示 SOAP 消息  
- Header：信封上的收件/快递单 → 安全、认证、路由、事务信息  
- Body：信封内容 → 业务请求/响应  
- Fault：投递失败通知 → 错误信息

**示例**

- Header 示例：

```xml
<soapenv:Header>
  <auth>
    <username>zhangsan</username>
    <password>123456</password>
  </auth>
</soapenv:Header>
```

- Body 示例：

```xml
<soapenv:Body>
  <getOrder>
    <orderId>1001</orderId>
  </getOrder>
</soapenv:Body>
```

- Fault 示例：

```xml
<soapenv:Fault>
  <faultcode>500</faultcode>
  <faultstring>服务器错误</faultstring>
</soapenv:Fault>
```

该结构可总结为：Envelope、Header、Body、Fault 四部分的组成与作用，便于记忆与复用。

#### 2.2.3 HTTP协议与SOAP的结合

**1、HTTP请求方法**：
| 请求方法 | 描述                                   |
| -------- | -------------------------------------- |
| POST     | 用于发送 SOAP 消息，带有请求体，最常用 |
| GET      | 一般不用于 SOAP，但可用于 WSDL 获取    |
| PUT      | 不常用，仅在特定资源更新时             |
| DELETE   | 一般不用于 SOAP                        |

2、<font color=brown>SOAP 通常基于 HTTP 传输，也就是说 SOAP = HTTP + XML</font>；SOAP 请求可能是 HTTP POST 或 HTTP GET 请求。
> PS：HTTP POST 请求规定至少两个 HTTP 头：Content-Type 和 Content-Length。

#### 2.2.4 SOAP网关
无论客户采用何种平台，只要发起请求，就会先将调用信息<font color=green>转换为 XML 格式</font>，SOAP 网关即可自动完成此转换。为确保参数、方法名与返回值的唯一性，SOAP 协议引入了私有标记表，使服务端的 SOAP 网关能够准确解析调用。同时，XML 作为编码表达，提供了更高层次的抽象能力，进而实现了平台与运行环境的透明性，真正做到跨平台与环境无关。

### 2.3 WSDL(Web Service Description Language)

#### 2.3.1 定义
1、WSDL 指网络服务描述语言
2、WSDL 使用 XML 编写
3、WSDL 是一种 XML 文档
4、WSDL 用于描述网络服务
5、WSDL 也可用于定位网络服务
6、这种文档可描述某个 Web service，它可规定服务的位置，以及此服务提供的操作（或方法）

简而言之，就是<font color=yellowgreen>定义了一套基于XML的语法，将Web服务描述为能够进行消息交换的服务访问点的集合</font>。

#### 2.3.2 主要元素
![WSDL主要元素](/img/SOA/WSDL主要元素.png)

#### 2.3.3 文档规范

``` xml 
<definitions namespace = “http://… ”>
	<types> XML schema types </type>
	<message>  definition of a message </message>
	<portType> a set of operations </portType>
	<binding> communication protocols </binding>
	<service> a list of binding and ports </service>
</definitions>
```

**WSDL概念模型**：

| 层次             | 元素                                                   | 性质         | 说明                                                                              |
| ---------------- | ------------------------------------------------------ | ------------ | --------------------------------------------------------------------------------- |
| 服务接口（抽象） | Message<br>PortType (WSDL 2.0：Interface)<br>Operation | 描述接口粒度 | Message 定义消息结构；PortType/Interface 定义端口类型；Operation 定义具体接口行为 |
| 绑定             | Binding                                                | 协议绑定     | 将抽象接口与传输协议（如 SOAP/HTTP）相关联，指定编码与数据交换格式                |
| 服务实现（部署） | Service<br>Port (WSDL 2.0：Endpoint)                   | 部署单元     | 1、Service 聚合 Binding 与端点；<br>2、Port/Endpoint 指向具体服务地址和通信协议   |

> PS：
> - 服务接口（抽象层）定义“服务契约”，聚焦可复用的接口规范与消息结构；
> - 绑定层定义“通信方式”，包括协议、传输与编码格式；
> - 服务实现（具体部署层）定义“可访问端点”，明确实际部署位置与协议配置。

### 2.4 UDDI（已被淘汰，没人用）
**定义：**
- UDDI 是一个独立于平台的框架，用于通过使用 Internet 来描述服务，发现企业，并对企业服务进行集成。
- UDDI 指的是通用描述、发现与集成服务
- UDDI 是一种用于存储有关 web services 的信息的目录。
- UDDI 是一种由 WSDL 描述的 web services 界面的目录。
- UDDI 经由 SOAP 进行通信
- UDDI 被构建入了微软的 .NET 平台

## 3. REST基础

### 3.1 REST基本概念

#### 3.1.1 REST定义
1、**基本含义**：REST是英文Representational State Transfer的缩写，<font color=purple>表象化状态转变</font> ，或者<font color=camel>表述性状态转移</font>
2、**拗口名字的由来**：
- **应用状态**（Application State）表示Web应用的客户端的状态，简单起见可以理解为<font color=red>会话状态</font>
- **资源**在浏览器中以**超媒体**的形式呈现
- 用户通过与超媒体交互推动状态转变 -> <font color=red>超媒体成为了驱动客户端会话状态的转换的引擎</font> -> 应用状态的转换体现为浏览器中呈现<font color=amber>资源的转换</font>
- 将超媒体进一步抽象成一般意义上的资源呈现（Representation ）方式，那么应用状态变成了可被呈现的状态（REpresentational State）。<font color=red>应用状态之间的转换就成了可被呈现的状态装换（REpresentational State Transfer）</font>

#### 3.1.2 关键特性

1、 **客户端-服务器（Client-Server）**  
  前端（浏览器/App）负责界面，后端负责业务与数据，二者分离、独立演化。

  **好处**：前后端解耦，更容易扩展与维护。

2、 **无状态性（Stateless）**  
  服务器不保存客户端会话状态。每次请求都需携带足够的信息让服务器理解。

  示例：

  - 传统会话（有状态）  
    客户端：我是谁？  
    服务器：我记住你了（Session）  
    客户端：我要查订单

  - REST（无状态）  
    客户端：我是 user=1001，这是我的 token，我要查订单  
    服务器：OK，返回订单

3、 **统一接口（Uniform Interface）**  
  REST 规定一套统一的操作方式，降低系统间协议复杂度：

  | 动作   | 描述 |
  | ------ | ---- |
  | GET    | 查询 |
  | POST   | 创建 |
  | PUT    | 更新 |
  | DELETE | 删除 |

4、 **资源可表述（Representation）**  
  资源与其表现形式分离，同一资源可返回不同格式（JSON、XML、HTML 等）。

  ```http
  GET /users/1001
  Accept: application/json
  ```

  响应：

  ```json
  {"id":1001,"name":"Alice"}
  ```

5、 **可缓存（Cacheable）**  
  服务器可标记响应为可缓存（如 `Cache-Control: max-age=3600`），浏览器或网关缓存后续请求，降低服务器压力。常见于图片、静态配置等不常变化的资源。

6、 **分层系统（Layered System）**  
  客户端无需关心是否直接访问应用服务器，或经过网关、负载均衡、代理。

  ```
  客户端
    ↓
  API Gateway
    ↓
  负载均衡
    ↓
  应用服务
  ```
7、按需编码

#### 3.1.3 无状态性
1、**定义**：服务器不保存客户端的会话信息（Session）

所以RESTful API每个请求都是独立的，彼此之间没有记忆。
客户端必须在每次请求中<font color=chocolate>携带身份信息</font>，例如：
- Token
- API Key
- JWT

2、对比有状态：
| 维度         | 有状态系统             | 无状态系统（REST）   |
| ------------ | ---------------------- | -------------------- |
| 会话         | 服务器保存 Session     | 服务器不保存 Session |
| 扩展性       | 差                     | **好**               |
| 负载均衡     | 困难                   | **容易**             |
| 可靠性       | 服务器崩溃会丢 Session | **更健壮**           |
| 客户端复杂度 | 低                     | 略高                 |

### 3.2 为什么使用REST？
1、**必要性**：
- API让web，手机客户端，桌面多种操作成为可能，程序员分工更加明确，切降低了开发成本；
- 软件开发依赖解耦
- 让编程语言发挥各自的优势。

2、**优势**：
- 将用户界面和数据存储分离，提高用户界面跨多个服务平台
- 轻松为不同的请求返回不同格式的数据
- 最佳的场景是对外提供公开的服务，也就是所谓的OpenAPI
- REST 的真正价值在于 Web Services，而不是通过浏览器操作的应用程序。

### 3.3 与MVC和RPC对比

#### 3.3.1 VS MVC

| 比较项     | MVC                                                | REST                                       |
| ---------- | -------------------------------------------------- | ------------------------------------------ |
| 关注点     | 服务器端逻辑分层与视图呈现；模型、视图、控制器解耦 | 统一资源接口，强调资源状态与表现形式的转换 |
| 平台与语言 | 依赖服务器端语言框架，跨语言解耦困难               | 跨平台、跨语言，可通过标准 HTTP/URI 实现   |
| 客户端关系 | 服务器端控制视图生成，客户端是延伸                 | 客户端直接消费资源，无须与服务器端视图耦合 |
| 接口风格   | 框架定义控制器方法，接口较隐式                     | 统一接口（HTTP 方法 + URI）清晰可见        |
| 典型示例   | Spring MVC、Jersey（也可支持 REST）                | 原生 RESTful 服务、ROA 架构                |

#### 3.3.2 VS RPC

| 比较项 | RPC                     | REST                         |
| ------ | ----------------------- | ---------------------------- |
| 模型   | 过程调用                | 资源视角                     |
| 协议   | 通常依赖专有协议或 SOAP | 基于标准 HTTP                |
| 表现   | 强调方法和参数          | 强调资源标识与状态           |
| 轻量级 | 较重，需封装与解析      | 较轻，直接利用 HTTP/JSON/XML |
| 可见性 | 方法和数据不直观        | URI 与 HTTP 方法明晰展现     |

## 4. RESTful API设计
### 4.1 请求的安全性和幂等性
#### 4.1.1 安全性和幂等性
- **安全性**是指外系统对该接口的访问，不会<font color=camel>使服务器资源的状态发生改变</font>
- **幂等性**是一个数学上的概念，在这里是指外系统对同一REST接口的多次访问，得到的资源状态是相同的。在网速不够快的情况下，客户端发送一个请求后不能立即得到响应，由于不能确定是否请求是否被成功提交，所以它有可能会再次发送另一个相同的请求，<font color=orange>幂等性决定了第二个请求是否有效</font>。

#### 4.1.2 四种方式的安全性和幂等性
| 请求方式 | 安全性 | 幂等性   | 说明                                                           |
| -------- | ------ | -------- | -------------------------------------------------------------- |
| GET      | ✅ 是   | ✅ 是     | 只读操作，不修改服务器状态，重复多次结果一致。                 |
| POST     | ❌ 否   | ❌ 通常否 | 创建或触发副作用，重复提交可能产生多份资源或状态变化。         |
| PUT      | ❌ 否   | ✅ 是     | 覆盖资源，内容相同重复请求结果一致，副作用可控。               |
| DELETE   | ❌ 否   | ✅ 是     | 删除资源，重复删除结果一致（资源不存在），但首请求会改变状态。 |
## 6. 服务组合技术

### 6.1 服务编排（Service Orchestration）

#### 6.1.1 服务编排的概念
**1、核心定义**

服务编排 = 中心化控制的服务调用方式

将多个小粒度的Web服务按照特定的业务逻辑规则构造为一个可执行的业务过程，同时又可以看作是一个大粒度的复合Web服务。

由**一个中央协调者**（Orchestrator）来统一控制多个服务的执行顺序、逻辑判断、异常处理和数据流转。

**2、形象化理解**：
你可以把它理解为：“导演模式”：有一个“总指挥”，所有服务都听它调度。
> PS: 侧重点在用已有的服务拼凑需求服务

#### 6.1.2 服务编排框架
![服务编排框架图](/img/SOA/服务编排框架图.png)

#### 6.1.3 BPEL(4WS)
BPEL能够实现基于WSDL的Web Services之间的流程编排和服务协同，它提供了一种XML注释和语义，用于指定对Web Services进行编排并确定Web服务之间的业务流程，实现Web Services之间的协同。

**基本结构**：
``` xml
1、过程中的基本功能单元：活动<activity>
2、活动之间的次序关系：
  1）先后次序<sequence>
  2）多分支<switch>
  3）循环<while>
  4）并发与同步<flow>
  5）非确定性选择<pick>
3、过程的相关数据：容器<container>
4、错误处理机制：<catching>、<handling faults>
5、补偿机制：<compensation handler>
```
#### 6.1.4 服务编排的运行模式
1、**集中式的执行引擎**（需要有一个中心控制引擎，负责调用各个离散的服务）

### 6.2 服务协同（Service Choreography）

#### 6.2.1 服务协同定义与框架
- 将多个零散的、分别由多方提供的服务/业务流程按照彼此之间的协同关系组织起来，支持多方的交互行为。
- 侧重于不同服务之间的消息传递的次序与规则，以保证期望的协同行为。
- 无需中心控制；
- 无需完全由一个组织所拥有；
![服务协同](/img/SOA/服务协同.png)

#### 6.2.2 WS-CDL（Web Service Choreography Description Language）

1、**定义**：一种描述“多个参与方如何交互”的<font color=plum>规范性</font>语言（是交互规范，不是程序）

它不关心谁控制流程，只关心：
- 谁给谁发消息
- 什么时候发
- 需要什么格式

可以理解为：<font color=green>“服务协同的蓝图”</font>


### 6.3 二者对比

## BPEL vs WS-CDL 对比
| 项目     | BPEL                 | WS-CDL                 |
| -------- | -------------------- | ---------------------- |
| 控制机制 | 需要集中式控制       | 无需集中式控制，分布式 |
| 服务组合 | 支持递归式服务组合   | n/a                    |
| 语言类型 | 可执行的过程建模语言 | 描述性语言，不可执行   |
| Web 服务 | 需要调用 Web 服务    | 无需调用 Web 服务      |
